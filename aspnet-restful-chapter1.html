Section 04

REST Architecture

Representational State Transfer is an architectural pattern for developing web services. REST communicates over HTTP, using its vocabulary — methods, URI syntax, status codes, and media types.

01

Client-Server

Pull-based interaction style. Client requests data from servers as and when needed.

02

Stateless

Each request must contain all the information necessary to understand it. No stored context on the server.

03

Cacheable

Responses must be labeled as cacheable or non-cacheable to improve network efficiency.

04

Uniform Interface

All resources accessed via a generic interface (HTTP GET, POST, PUT, DELETE) using URIs.

05

Named Resources

The system is comprised of resources named using URLs, interconnected via hyperlinks.

Section 06

Introducing ASP.NET Core

ASP.NET Web API is a service-based framework for building RESTful services. It is based on MVC minus the View, optimized for headless services. API calls use HTTP verbs over a Uniform Resource Identifier (URI).

Cross-Platform

Run on Windows, Linux, and macOS. Ideal for microservices that mix technologies across service boundaries.

Docker Ready

The modularity and lightweight nature of .NET makes it ideal for containers and microservices architectures.

High Performance

ASP.NET Core is a top-performing web framework on TechEmpower benchmarks for both Windows and Linux.

Section 08

HTTP Verbs in ASP.NET Core

The four core verbs map directly to CRUD operations. Click each to explore.

GET — Read

Used to retrieve a specific resource by its identity or a collection of resources. GET has no side effects — it only reads data.

SafeIdempotentNo Body
[HttpGet]
public ActionResult<IEnumerable<Product>> GetAll()
{
    return Ok(_repo.GetAll());
}

POST — Create

Used to generate or create a new resource. Data is passed in the body as an entity object. Returns 201 Created on success.

Non-SafeNon-IdempotentHas Body
[HttpPost]
[ProducesResponseType(StatusCodes.Status201Created)]
[ProducesResponseType(StatusCodes.Status400BadRequest)]
public ActionResult<Product> Create(Product product)
{
    _repo.Add(product);
    return CreatedAtAction(nameof(GetById), new { id = product.Id }, product);
}

PUT — Update

Used to update an existing resource by its identity. Can send data as an object or parameter. Replaces the full resource.

Non-SafeIdempotentHas Body
[HttpPut("{id}")]
public IActionResult Update(int id, [FromBody] Product product)
{
    if (!_repo.Exists(id)) return NotFound();
    _repo.Update(id, product);
    return NoContent();
}

DELETE — Remove

Used to delete an existing resource by its identity. Can send data as an object or parameter in the route.

Non-SafeIdempotent
[HttpDelete("{id}")]
public IActionResult Delete(int id)
{
    if (!_repo.Exists(id)) return NotFound();
    _repo.Delete(id);
    return NoContent(); // 204
}

ASP.NET Core transforms error results (status 400+) into responses with ProblemDetails for standardized error format.

Section 10

Binding Source Parameter Inference

A binding source attribute defines where an action parameter's value is found. ASP.NET Core infers these automatically in most cases.

AttributeBinding Source
[FromBody]Inferred for complex types. Only one FromBody can exist per action.
[FromForm]Inferred for IFormFile and IFormFileCollection — uses multipart/form-data.
[FromHeader]Reads from request headers.
[FromQuery]Inferred for any other action parameters — reads from query string.
[FromRoute]Inferred for any parameter matching a route token name.
[FromServices]Injects the request service as an action parameter.
 
// Example: combined binding sources
[HttpGet("{id}")]
public ActionResult<Product> GetProduct(
    [FromRoute] int id,         // from /products/42
    [FromQuery] string lang,    // from ?lang=en
    [FromHeader] string token   // from Authorization header
) { ... }
